#### `ProjectCard` — `impact` prop 타입 선언과 실제 사용 불일치 🟠
**File**: `app/page.tsx` | **Line(s)**: 83-97, 301

**Problem**:
`ProjectCard` 컴포넌트의 `impact` prop 타입이 `string`으로 선언되어 있습니다. 호출부에서 `project.impact[0]`으로 첫 번째 항목만 전달하므로 현재는 런타임 오류가 없지만, `impact` 배열이 비어 있으면 `undefined`가 전달되어 `string` 타입 계약을 위반합니다. TypeScript 가 strict null check 환경에서 이를 잡지 못하는 이유는 배열 인덱스 접근이 `noUncheckedIndexedAccess` 없이는 항상 `T`로 추론되기 때문입니다.

**Current Code**:
```tsx
function ProjectCard({
  ...
  impact,
}: {
  ...
  impact: string;   // 선언: string
  ...
}) { ... }

// 호출부
impact={project.impact[0]}   // impact 배열이 빈 경우 undefined 전달 가능
```

**Recommended Fix**:
```tsx
// 옵션 A: 타입을 실제 사용에 맞게 조정
impact: string | undefined;
// ...
{impact && (
  <p className="mt-3 text-sm font-medium leading-7 text-primary">{impact}</p>
)}

// 옵션 B: 빈 배열 방어
impact={project.impact[0] ?? ""}
```

**Rationale**: 현재 데이터에선 영향이 없지만, 새 프로젝트를 추가할 때 `impact` 배열을 비워두면 조용히 렌더링 오류가 발생할 수 있습니다. 타입과 런타임 동작을 일치시켜 두면 이후 콘텐츠 추가 시 안전합니다.

#### `philosophy` 인덱스가 9를 초과할 경우 두 자리 숫자 표시 오류 🟡
**File**: `app/page.tsx` | **Line(s)**: 548

**Problem**:
철학 항목의 순번을 `` `0${index + 1}` `` 로 포매팅합니다. 항목이 9개 이하이면 `01`~`09`로 표시되지만, 10개 이상이면 `010`이 됩니다. 현재는 4개이므로 문제없으나 향후 확장 시 버그가 됩니다.

**Current Code**:
```tsx
<p className="font-mono text-[11px] font-medium text-accent-ink">
  0{index + 1}
</p>
```

**Recommended Fix**:
```tsx
<p className="font-mono text-[11px] font-medium text-accent-ink">
  {String(index + 1).padStart(2, "0")}
</p>
```

**Rationale**: 동일한 패턴이 트레이드오프 번호(`line 374`)에도 있으니 두 곳 모두 수정하는 것이 좋습니다.

### 렌더링 중복

#### DetailBlock 성과 목록과 성과 카드가 중복 렌더링됨 🟡
**File**: `app/page.tsx` | **Line(s)**: 330-340, 358-363

**Problem**:
프로젝트 상세 블록(DetailBlock)에서 `impact` 배열이 두 번 렌더링됩니다.

1. 왼쪽 패널에서 `project.impact` 전체를 "핵심 성과 1", "핵심 성과 2" 카드로 반복(line 330-340)
2. 오른쪽 DetailBlock "성과" 섹션에서 `project.impact` 전체를 또 반복(line 358-363)

같은 데이터를 같은 뷰 내에서 두 번 보여주면 콘텐츠가 중복되어 가독성을 해치고, 화면 공간을 낭비합니다.

**Current Code**:
```tsx
{/* 왼쪽 패널 */}
{project.impact.map((i, idx) => (
  <div key={`impact-${idx}`} className="mt-8 ...">
    <p>핵심 성과 {idx + 1}</p>
    <p>{i}</p>
  </div>
))}

{/* 오른쪽 DetailBlock */}
<DetailBlock label="성과">
  <ul>
    {project.impact.map((item) => (
      <li key={item}>{item}</li>
    ))}
  </ul>
</DetailBlock>
```

**Recommended Fix**:
왼쪽 패널의 성과 카드는 `impact[0]`(가장 중요한 성과 1건)만 하이라이트하고, 오른쪽 DetailBlock에 전체 목록을 유지합니다.

```tsx
{/* 왼쪽 패널 — 대표 성과 1건만 */}
{project.impact[0] && (
  <div className="mt-8 rounded-[1.5rem] border border-strong bg-inset-emphasis p-5">
    <p className="font-mono text-[11px] font-medium text-accent-ink">핵심 성과</p>
    <p className="mt-3 text-base font-medium leading-7 text-primary">
      {project.impact[0]}
    </p>
  </div>
)}

{/* 오른쪽 DetailBlock — 전체 목록 유지 */}
```

**Rationale**: 좌측은 임팩트가 큰 대표 성과 1건을 시각적으로 강조하고, 우측은 상세 정보를 제공하는 분리된 역할을 갖도록 하면 정보 위계가 명확해집니다.

### 시맨틱 / 접근성

#### `<pre>` 태그를 레이아웃 컨테이너로 오용 🟠
**File**: `app/page.tsx` | **Line(s)**: 593-596

**Problem**:
GitHub 링크 카드 내부에서 `<pre>` 요소를 flexbox 레이아웃 컨테이너로 사용합니다. `<pre>`는 HTML 시맨틱상 "사전 포매팅된 텍스트(preformatted text)"를 의미하며, 스크린리더는 이 안의 내용을 고정폭 텍스트 블록으로 읽을 수 있습니다. 또한 기본 CSS(`white-space: pre`)가 Tailwind 스타일과 충돌할 가능성이 있습니다.

**Current Code**:
```tsx
<pre className="flex self-stretch items-center gap-2">
  <Image src={link.img} width={40} height={40} alt={'github-logo'} />
  <p className="text-sm font-semibold text-ink">Go to {link.label}</p>
</pre>
```

**Recommended Fix**:
```tsx
<div className="flex self-stretch items-center gap-2">
  <Image src={link.img} width={40} height={40} alt="github-logo" />
  <p className="text-sm font-semibold text-ink">Go to {link.label}</p>
</div>
```

**Rationale**: `<div>`로 교체하면 시맨틱 오용이 없어지고 `white-space: pre` 기본값으로 인한 잠재적 레이아웃 버그도 제거됩니다.

#### Image `alt` 속성이 하드코딩되어 동적 링크 확장 시 깨짐 🟡
**File**: `app/page.tsx` | **Line(s)**: 594

**Problem**:
`alt={'github-logo'}`로 하드코딩되어 있어, `content.contact.links`에 GitHub 외 다른 링크(예: LinkedIn)가 추가되면 alt 텍스트가 잘못됩니다.

**Current Code**:
```tsx
<Image src={link.img} width={40} height={40} alt={'github-logo'} />
```

**Recommended Fix**:
```tsx

```

**Rationale**: `link.label`을 동적으로 참조하면 링크가 추가되더라도 alt 텍스트가 항상 올바릅니다.
