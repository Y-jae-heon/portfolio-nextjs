## 조정 대상: Hero — headline + intro (REQ-002)

### Before

```typescript
headline: "복잡한 서비스 문제를\n구조적으로 해결하고,\n실제 운영 가능한 시스템으로 만듭니다.",

intro: "프론트엔드 Web and App, 서버 아키텍처, AI 기능까지 End-to-End로 연결합니다. 모호한 요구를 구조화하고, 제품으로 빠르게 실행하는 데 강점이 있습니다.",
```

**문제점:**
- headline이 일반적인 "문제 해결사" 서사에 머물러 있어 차별화 포인트가 없음
- intro가 기술 스택 나열과 추상적 강점 설명에 그침. 이력서에서 반복 등장하는 핵심 태도("도구가 없으면 직접 만든다", "역할이 없으면 채운다")가 드러나지 않음
- "Web and App"이라는 어색한 표현 존재

---

### After

```typescript
headline: "주어진 도구로 안 되면 직접 만들고,\n팀에 비어 있는 역할이 있으면\n채워왔습니다.",

intro: "React·React Native 기반 B2B·B2C 서비스를 5년 넘게 리딩하면서, 서드파티 한계에 부딪히면 Android Native 모듈을 직접 개발해 브릿징하고, 타이머 오차가 반복되면 브라우저 렌더링 사이클부터 다시 짚었습니다.",
```

---

### 근거

**이력서 대조 결과:**
- headline은 CLAUDE.md에 "이 이력서에 담긴 염재헌은 문제 앞에서 멈추지 않는 사람이에요... 이 패턴이 이력서 전반에 반복돼요. 한 번의 우연이 아니라 일관된 태도예요."로 명시. 이 태도를 직접 드러내는 문장.
- intro의 두 가지 사례(Android Native 모듈 브릿징, requestAnimationFrame 타이머)는 이력서 슬로그업 섹션에서 각각 SMTOWN과 마인드아너스 프로젝트의 핵심 기술로 명시됨.
- "5년 넘게"는 이력서 경력 5년 6개월(2020.06 ~ 현재)에서 도출. 반올림하지 않고 "넘게" 표현으로 정확성 유지.

**톤 비교:**
- Before: 범용적 시니어 개발자 포지셔닝. 어디서나 볼 수 있는 문구.
- After: 구체적인 행동 패턴("직접 만든다", "채워왔다")으로 개성 있는 첫 인상 형성. 프로젝트 섹션 내용과 자연스럽게 연결.

---

### 규칙 검증

- [x] CONTENT_ONLY: `app/content.ts`의 `content.headline`, `content.intro` 필드만 수정
- [x] RESUME_ALIGNED: headline 문장과 intro 사례가 이력서 내용에 직접 기반
- [x] NO_FABRICATION: "5년 넘게"는 실제 경력 기간(2020.06 ~ 2026.03 = 약 5년 9개월)에서 도출

---

### 연쇄 영향

- **chips 배열**: 현재 `["End-to-End 개발", "서비스 아키텍처 설계", "운영 문제 해결", "AI 기능 설계 및 적용"]`. headline 변경 후 chips도 "멀티플랫폼 연동", "서비스 리딩" 방향으로 1~2개 교체를 별도 검토할 수 있으나 P0 필수 범위 밖.
- **summary.lead**: 현재 문구와 방향이 겹치지 않으므로 별도 수정 불필요.
- **page.tsx**: headline은 `\n`으로 줄바꿈 처리되어 있으므로 렌더링 로직 변경 불필요. 기존 `whitespace-pre-line` 스타일 확인 권장.

## 조정 대상: Hero — heroMetrics (REQ-003)

### Before

```typescript
heroMetrics: [
  {
    value: "0→1",
    label: "SaaS 플랫폼 및 AI 서비스 구축",
    detail: "자동 생성형 SaaS와 AI 기능(OCR·LLM)을 실제 서비스로 설계하고 운영 단계까지 연결",
  },
  {
    value: "Multi",
    label: "20+ 프로젝트 수행",
    detail: "웹, 모바일, IoT, AI까지 다양한 도메인에서 서비스 구축 및 문제 해결 경험",
  },
  {
    value: "Lead",
    label: "기술 리딩 및 구조 개선",
    detail: "기술전략팀 리딩, 업무 워크플로우 개선, 세션 전반적인 시스템 개선 등 복잡한 문제 해결 주도",
  },
],
```

**문제점:**
- "0→1", "Multi", "Lead"는 구체적이지 않아 임팩트가 약함
- `value` 필드가 숫자/수치가 아닌 추상 단어로 채워져 있어 스캐너빌리티 낮음
- Boostree, DoctorInHome 기반 성과를 반영하고 있어 projects 교체 시 정합성 깨짐
- 가장 강력한 정량 성과(리뷰 분석 80% 단축)가 어디에도 없음

---

### After

```typescript
heroMetrics: [
  {
    value: "80%",
    label: "AI 파이프라인으로 분석 시간 단축",
    detail: "OpenAI + TensorFlow 기반 리뷰 분석 자동화로 수동 작업 시간 80% 절감 (리뷰핀치)",
  },
  {
    value: "5년+",
    label: "React / React Native 기반 B2B·B2C 서비스 리딩",
    detail: "20개 이상의 프로젝트 수행, SaaS·AI·IoT·모바일 도메인 End-to-End 경험",
  },
  {
    value: "Native",
    label: "Android 네이티브 모듈 직접 개발",
    detail: "서드파티 한계를 Android Native Module + RN 브릿징으로 직접 돌파한 경험 (SMTOWN)",
  },
],
```

---

### 근거

**이력서 대조 결과:**
- `80%`: 이력서 리뷰핀치 항목에 "AI 파이프라인 구축으로 분석 시간 80% 단축" 명시. 포트폴리오 전체에서 가장 강력한 단일 정량 수치.
- `5년+`: 이력서 경력 기간 2020.06 ~ 현재 = 약 5년 9개월. "20개 이상의 프로젝트"는 이력서 내 프로젝트 수 집계에 근거.
- `Native`: 이력서 SMTOWN 항목에 "Android Native 모듈 직접 개발" 명시. 단순 사용이 아닌 직접 개발이라는 점이 차별점.

**배치 논리:**
- 첫 번째(80%): 가장 강력한 수치로 시선 먼저 확보
- 두 번째(5년+): 경력 신뢰도 확립
- 세 번째(Native): 기술 깊이 증명. 역량(capabilities) 섹션 "멀티플랫폼 연동"과 맥락 연결

**projects 정합:**
- `80%` detail에 "(리뷰핀치)" 명시 → REQ-001에서 reviewpinch 프로젝트와 연결됨
- `Native` detail에 "(SMTOWN)" 명시 → REQ-001에서 smtown 프로젝트와 연결됨

---

### 규칙 검증

- [x] CONTENT_ONLY: `app/content.ts`의 `content.heroMetrics` 배열만 수정
- [x] RESUME_ALIGNED: 모든 수치와 설명이 이력서 명시 내용에 기반
- [x] NO_FABRICATION: "80%"는 이력서 리뷰핀치 항목, "5년+"은 경력 기간, "Native"는 SMTOWN 항목에서 직접 인용

---

### 연쇄 영향

- **REQ-001 Projects**: `heroMetrics`의 `80%`(리뷰핀치)와 `Native`(SMTOWN)가 projects 섹션과 수치·문맥이 정합되어야 함. 두 티켓 동시 적용 필요.
- **page.tsx**: `HeroMetric` 타입(`value`, `label`, `detail`) 변경 없음. 렌더링 로직 변경 불필요.
- **chips 배열**: "AI 기능 설계 및 적용" chip이 metrics의 `Native` 방향과 다소 충돌할 수 있으나 P0 범위 밖. 별도 검토 가능.
